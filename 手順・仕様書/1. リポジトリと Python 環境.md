# 1. リポジトリと Python 環境（詳細手順）

本書は `作成手順.md` の「1. リポジトリと Python 環境」を、実際に手を動かせる粒度でまとめたものです。対象は `仕様書.md` に基づくブラックジャック Web アプリ（Django + PostgreSQL + Docker）です。

**Git の流れは、GitHub でリポジトリを作成するところから記載しています。**

---

## 1.1 この章で行うことの目的

- **GitHub リポジトリ**: リモートに本体を置き、バックアップ・共有・履歴管理の基点にする。
- **ローカル Git**: 変更をコミットし、GitHub と `push` / `pull` で同期する。
- **Python 仮想環境**: このプロジェクト専用のライブラリ置き場を作り、他プロジェクトや OS の Python と衝突させない。
- **requirements.txt**: 使うパッケージとバージョンを記録し、同じ環境を再現しやすくする。

---

## 1.2 前提

- **GitHub アカウント**を持っていること（なければ https://github.com で無料登録）。
- Python 3.10 以上が `python3` または `python` で起動できること。
- ターミナルでコマンドを実行できること。

本プロジェクトの GitHub リポジトリは **nakaokazuki/blackjack** です。確定 URL はプロジェクトルートの `リポジトリURLメモ.md` にも記載しています。

---

## 1.3 GitHub で新規リポジトリを作成する

1. ブラウザで https://github.com にログインする。
2. 右上の **「+」** → **「New repository」** を選ぶ。
3. **Repository name** に例: `blackjack` を入力する。
4. **Public** / **Private** を選ぶ（誰に見せるかの違い。練習ならどちらでも可）。
5. **「Initialize this repository with」** の扱い（重要）:
   - **すでにローカルに `仕様書.md` などファイルがある**場合は、**README・.gitignore・ライセンスにチェックを入れない**空のリポジトリにすると、`git push` が素直です。
   - **README ありで作成した**場合は、後述「1.7」のとおり、初回 `push` 前にリモートのコミットと履歴を合わせる手順が必要になります。
6. **「Create repository」** をクリックする。

作成後、画面に表示される **リポジトリの URL** を控える。

- HTTPS: `https://github.com/nakaokazuki/blackjack.git`
- SSH: `git@github.com:nakaokazuki/blackjack.git`（SSH 鍵を GitHub に登録済みの場合）

---

## 1.4 ローカルにプロジェクトフォルダを用意する（どちらか一方）

### パターン A: 空の GitHub リポジトリを clone する（おすすめ）

これからフォルダを新規に作る場合:

```bash
cd /Users/kazuki/Desktop
git clone https://github.com/nakaokazuki/blackjack.git
cd blackjack
```

この `blackjack` フォルダが **プロジェクトルート**です。以降のコマンドは基本的にここで実行します。

### パターン B: すでにあるフォルダを GitHub とつなぐ

例: `/Users/kazuki/Desktop/blackjack` に既にファイルがある場合:

```bash
cd /Users/kazuki/Desktop/blackjack
```

まだ `git init` していなければ、次章で初期化します。

---

## 1.5 ローカルで Git を初期化する（パターン B のみ）

パターン A（`git clone` 済み）の場合は **すでに `.git` があるため不要**です。

パターン B で、まだ Git を使っていない場合:

```bash
git init
```

デフォルトブランチ名を `main` にそろえておくと GitHub と合わせやすいです。

```bash
git branch -M main
```

---

## 1.6 `.gitignore` の用意

Python・Django でよく無視するものを登録します。プロジェクトルートに `.gitignore` を作成し、例えば次を含めます。

- 仮想環境フォルダ: `.venv/` または `venv/`
- Python のキャッシュ: `__pycache__/`, `*.py[cod]`
- ローカル設定: `.env`（秘密情報を書く場合）
- Django: `*.log`, `db.sqlite3`（使う場合）, `staticfiles/` や `media/`（収集・アップロード先によって）
- IDE: `.idea/`, `.vscode/`（チーム方針に合わせる）

`.gitignore` はプロジェクトの進行に合わせて追記してよいです。

---

## 1.7 初回コミットと GitHub へ push

### 共通: 初回コミット

プロジェクトルートで:

```bash
git add .
git status   # コミット対象を確認（.venv や .env が含まれていないか）
git commit -m "Initial commit: project docs and Python environment setup"
```

### リモート `origin` を登録（パターン B）

`clone` していない場合は、GitHub に表示されている URL で追加します。

```bash
git remote add origin https://github.com/nakaokazuki/blackjack.git
```

（SSH を使う場合は `git@github.com:nakaokazuki/blackjack.git`）

### push

```bash
git push -u origin main
```

初回のみ `-u origin main` とすると、以降は `git push` でよくなります。

### GitHub 作成時に README などを付けていた場合

リモートに先にコミットがあり、`push` が拒否されることがあります。そのときは例えば:

```bash
git pull origin main --rebase
# または履歴を合わせる場合: git pull origin main --allow-unrelated-histories
git push -u origin main
```

競合が出たらファイルを編集してから `git add` → `git commit` → `git push` します。

## 1.8 Python のバージョン確認

```bash
python3 --version
```

表示が `Python 3.10.x` 以上であれば、この手順の前提を満たします。  
`python` だけで 3.x が動く環境なら、`python --version` で同様に確認してください。

---

## 1.9 仮想環境の作成

プロジェクトルートで実行します。

```bash
python3 -m venv .venv
```

- **`.venv`**: 仮想環境のディレクトリ名の例です。`venv` など別名でも構いません。
- 作成後、プロジェクトルートに `.venv` フォルダが現れます。

**注意**: `.venv` は Git に含めないように `.gitignore` に書いておきます。

---

## 1.10 仮想環境の有効化

### macOS / Linux（bash や zsh）

```bash
source .venv/bin/activate
```

プロンプトの先頭に `(.venv)` のように環境名が付けば有効化できています。

### 無効化するとき

```bash
deactivate
```

### Windows（参考）

コマンドプロンプト:

```text
.venv\Scripts\activate.bat
```

PowerShell:

```text
.venv\Scripts\Activate.ps1
```

## 1.11 pip の更新（任意・推奨）

仮想環境を有効化した状態で:

```bash
python -m pip install --upgrade pip
```

これにより、以降の `pip install` が安定しやすくなります。

---

## 1.12 `requirements.txt` の作成

プロジェクトルートに `requirements.txt` を置き、**仕様に必要な最低限**の例は次のとおりです。

```text
Django>=4.2,<5.2
psycopg2-binary>=2.9
```

- **Django**: LTS を使う場合は 4.2 系、新しめを試す場合はプロジェクト方針に合わせて上限を変えてください。
- **psycopg2-binary**: PostgreSQL 接続用。本番でソースビルドする方針なら `psycopg2` に切り替える判断もあります（別途ビルド要件が必要）。

バージョンを**ピン留め**（再現性を高める）する例:

```text
Django==4.2.17
psycopg2-binary==2.9.10
```

実際の番号は `pip install` 後に `pip freeze` で確認して更新してもよいです。

---

## 1.13 依存パッケージのインストール

仮想環境が有効な状態で、プロジェクトルートから:

```bash
pip install -r requirements.txt
```

エラーなく完了すれば、この環境で Django と PostgreSQL ドライバが使える状態です。

---

## 1.14 動作確認（任意）

```bash
python -c "import django; print(django.get_version())"
python -c "import psycopg2; print('psycopg2 OK')"
```

バージョン番号や `psycopg2 OK` が表示されれば、インストールは成功しています。

---

## 1.15 よくあるつまずき

| 現象 | 対処の例 |
|------|-----------|
| `python3: command not found` | Python 3 のインストールパスを確認する。macOS では `pyenv` や公式インストーラで 3.x を入れる。 |
| `venv` 作成で権限エラー | プロジェクトを書き込み可能なディレクトリに置く。 |
| `pip install` が別の Python に入っている | 仮想環境を `source .../activate` したか確認する。`which python` で `.venv` 配下を指すか確認。 |
| `psycopg2-binary` のインストール失敗 | 古い pip を `python -m pip install -U pip` で更新して再試行。それでもダメなら Python のバージョンと OS を確認。 |
| `git push` が拒否される | リモートに README だけの先コミットがある場合は「1.7」の `git pull` を試す。認証エラーなら GitHub の Personal Access Token（HTTPS）または SSH 鍵を確認。 |
