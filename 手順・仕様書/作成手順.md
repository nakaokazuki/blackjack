# ブラックジャック Web アプリ 作成手順

本手順は `仕様書.md`（v1.1）に従って実装を進めるためのものです。  
**Django・PostgreSQL・Docker はローカルにインストール済み**である前提とし、ダウンロード手順は省略します。

**GitHub リポジトリ**: [https://github.com/nakaokazuki/blackjack](https://github.com/nakaokazuki/blackjack)（ルートの `リポジトリURLメモ.md` にも記載）

---

## 0. 事前確認

- Python 3.10 以上（プロジェクトで採用する版に合わせる）が利用できること
- `docker` / `docker compose`（または `docker-compose`）が動くこと
- `psql` は必須ではないが、DB 接続トラブル時の確認に便利

---

## 1. リポジトリと Python 環境

1. 作業ディレクトリを `blackjack`（本プロジェクトルート）にする。
2. 仮想環境を作成し有効化する（例: `python -m venv .venv` → `source .venv/bin/activate`）。
3. 依存パッケージを固定するため `requirements.txt` を用意する。最低限の例:
   - `Django`（LTS またはプロジェクト方針の版）
   - `psycopg2-binary`（PostgreSQL 用ドライバ）

---

## 2. Docker Compose（Web + PostgreSQL）

仕様書「3. システム構成」に合わせ、**アプリ用コンテナ**と **PostgreSQL コンテナ**を同一ネットワークで起動できるようにする。

1. プロジェクトルートに `Dockerfile`（Django アプリ用）を置く。
2. `docker-compose.yml` を置き、少なくとも次を定義する。
   - **db**: 公式 PostgreSQL イメージ、`POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB`、ボリューム（データ永続化）
   - **web**: 上記 `Dockerfile` からビルド、`depends_on: db`、環境変数で DB 接続情報を渡す、`8000:8000` 等でポート公開
3. 仕様書 10 章の未決定項目をここで決める。
   - サービス名（例: `web`, `db`）
   - ホスト側ポート（例: Web `8000`, DB `5432` は公開しない運用も可）
   - Django から読む変数（例: `POSTGRES_HOST=db`, `POSTGRES_PORT=5432`, `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`）

**注意**: ローカルで `runserver` のみ使い DB だけ Docker、という構成も可能。いずれにせよ Django の `DATABASES` は PostgreSQL を指す。

---

## 3. Django プロジェクト作成

1. `django-admin startproject config .` のように、設定パッケージ名を決めてプロジェクトを生成する（`manage.py` がルートに来る構成が扱いやすい）。
2. アプリを 1 つにまとめるか分割するかは任意。例: `python manage.py startapp game` でゲーム・画面・モデルをまとめる、または `accounts` と `game` に分ける。
3. `config/settings.py`（または分割設定）で次を行う。
   - `INSTALLED_APPS` に自作アプリと `django.contrib` 系を登録
   - **PostgreSQL** の `DATABASES`（環境変数から読み込み推奨）
   - `LANGUAGE_CODE` / `TIME_ZONE`（日本向けなら `ja` / `Asia/Tokyo` 等）
   - テンプレート・静的ファイルのディレクトリ設定
   - `LOGIN_URL` / `LOGIN_REDIRECT_URL` / `LOGOUT_REDIRECT_URL`（未ログイン時の誘導に使用）

---

## 4. ユーザーとポイント（データモデル）

仕様書「4.1」「4.2」「7. データモデル」に合わせる。

1. **ポイント**をユーザーに紐づける。実装パターンはいずれか。
   - `AbstractUser` を継承したカスタムユーザーモデルに `points`（整数、**初期値 0**）を追加
   - または `User` と `OneToOne` のプロファイルモデルに `points` を持たせる
2. `AUTH_USER_MODEL` をカスタムユーザーに設定する場合は、**マイグレーション前**に決める。
3. `python manage.py makemigrations` → `migrate` を実行する。
4. スーパーユーザー作成（管理画面を使う場合）: `createsuperuser`

**ポイント増減の定数**（仕様書 4.2・5 章）をコード上で明示する。

- 勝ち: `+10`
- 負け: `-10`
- 引き分け: `0`

仕様書 10 章の「負のポイントを許可するか」は実装前に決め、ビジネスルールとして `clean` やゲーム終了処理に反映する。

---

## 5. 認証（登録・ログイン・ログアウト）

仕様書「4.1」「6.3」「8.」に合わせる。

1. **ユーザー登録**: `UserCreationForm` を拡張するか、`ModelForm` でユーザー名・パスワードを受け取り、`points=0` で保存。パスワードは Django 標準のハッシュ保存のみ。
2. **ログイン / ログアウト**: `LoginView` / `logout`（関数ビューまたは `LogoutView`）を用意。
3. **Game 画面およびゲーム用 API** には `@login_required` または `LoginRequiredMixin` を適用し、未認証は **ログインまたは登録画面へリダイレクト**（仕様書 4.3）。

---

## 6. URL 設計と画面ルーティング

仕様書「6. 画面仕様」に対応するパスを決める（例）。

| 画面 | パス例 | 内容 |
|------|--------|------|
| Top | `/` | ゲームスタート、ユーザー登録導線、ログイン中表示、ポイント表示 |
| Game | `/game/` | ブラックジャック本体（要ログイン） |
| User | `/user/` または `/accounts/` | 登録・ログイン・ログイン中はユーザー情報表示 |

`config/urls.py` でルートを include し、名前付き URL（`{% url %}`）でテンプレートから参照できるようにする。

---

## 7. テンプレート共通要件

1. ベーステンプレート（ヘッダ・ナビ・メッセージ表示）を 1 枚用意し、Top / Game / User で継承する。
2. ログイン状態は `user.is_authenticated` で分岐し、仕様どおり文言・ポイントを表示する。
3. CSRF はフォームに `{% csrf_token %}` を必須とする。

## 10. ブラックジャックのドメインロジック

仕様書「5.」に従い、**ビューに全部書かず**関数またはクラスに切り出す。

1. **カード**: スート・ランク（トランプ 52 枚）。内部表現はタプルや小さな dataclass でよい。
2. **デッキ**: 1 デッキまたは複数デッキをシャッフル（`random.shuffle`）。未使用カードをスタックとして保持。
3. **手札の合計**: A を 1/11 の有利な方で数える一般的なルール。仕様書 10 章の「ソフト 17」はディーラー側の閾値として決め、コードと仕様書に追記する。
4. **初期配布**: プレイヤー 2 枚、ディーラー 2 枚。UI ではディーラーの裏札を表現（セッションまたはサーバー側状態に「表向き／裏向き」を保持）。
5. **プレイヤー操作**: POST 等で **ヒット** / **スタンド**のみ受け付ける。
6. **スタンド後**: ディーラーが **17 以上**になるまでヒット（閾値は仕様追記で固定）。
7. **勝敗**: バースト、ブラックジャック（必要なら自然ブラックジャックの優先）、同点は引き分け。
8. **終了処理**: 結果に応じてユーザーの `points` を更新（**+10 / -10 / 0**）。トランザクション（`transaction.atomic`）で更新と整合性を保つ。
9. **ゲーム状態の保持**: セッションキーで「進行中のデッキ・手札」を保存するか、サーバー側の一時テーブル／キャッシュを使うかを決める。単一サーバー・開発用途ならセッションで足りることが多い。
