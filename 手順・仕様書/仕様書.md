# ブラックジャック Web アプリケーション 仕様書

## 1. 概要

Django・PostgreSQL・Docker を用いて、ユーザー登録・ログインが可能なブラックジャックゲームを提供する。各ユーザーはポイントを保有し、画面上で確認できる。対戦相手はコンピューター（ディーラー）とする。

## 2. 技術スタック

| 区分 | 技術 |
|------|------|
| アプリケーション | Django |
| データベース | PostgreSQL |
| コンテナ | Docker（および Docker Compose による複数サービス構成を想定） |

## 3. システム構成（想定）

- **Web**: Django アプリケーション（HTTP サーバーは開発時は `runserver`、本番は gunicorn 等を想定可）
- **DB**: PostgreSQL（Django の `DATABASES` で接続）
- **Docker**: アプリコンテナと DB コンテナを分離し、同一ネットワーク上で連携

詳細な `Dockerfile` / `docker-compose.yml` の構成は実装フェーズで定義する。

## 4. 機能要件

### 4.1 ユーザー

- ユーザー名・パスワードによる**新規登録**ができる。
- **ログイン**・**ログアウト**ができる。
- ログイン中は**ユーザー情報**（少なくともユーザー名、保有ポイント）を画面で確認できる。

### 4.2 ポイント

- **ユーザーごとにポイント**を保持する。**初期保有ポイントは 0** とする。
- ゲームの勝敗に応じてポイントが増減する。**勝ち +10、負け -10、引き分け 0**（変動なし）とする。
- Top 画面およびユーザー関連画面で、ログイン中ユーザーに対し**現在のポイントが表示**される。

### 4.3 ゲーム（ユーザー vs コンピューター）

- **ログイン中のユーザーのみ**プレイ可能とする。未ログインの場合はゲーム開始・Game 画面アクセスを拒否し、ログインまたはユーザー登録へ誘導する。
- **プレイヤー**（ログインユーザー）と**ディーラー**（コンピューター）で 1 手ずつ進行する標準的なブラックジャックとする。
- 1 ゲーム終了後、結果に応じてポイントを更新し、**Top 画面へ遷移**する。

## 5. ブラックジャックのルール（アプリ内仕様）

以下は実装の基準とする。細部（例: ソフト 17 の扱い）は実装時に本仕様書へ追記してよい。

- **カード・デッキ**: **トランプ**（標準の 52 枚、4 スート × 13 ランク）を用いる。使用デッキ数（1 デッキ / 複数デッキのシャッフル）は実装で決定してよい。
- **初期配布**: プレイヤー・ディーラーにそれぞれ 2 枚（ディーラーの 1 枚は表向き・もう 1 枚は裏向きなど、一般的な表示を踏襲）。
- **プレイヤーの操作**: **ヒット**（カードを 1 枚追加）と**スタンド**（これ以上引かない）**のみ**とする。ダブルダウン、スプリット、サレンダー等は採用しない。
- **ディーラー**: プレイヤーがスタンドした後、規則に従いカードを引く（例: 17 以上になるまでヒット）。
- **勝敗判定**: バースト（22 超）、ブラックジャック、通常の合計比較による勝ち・負け・引き分け。
- **ポイント増減**（1 ゲームあたり）: **勝ち +10、負け -10、引き分け 0**。

## 6. 画面仕様

### 6.1 Top（トップ）

| 項目 | 内容 |
|------|------|
| 目的 | アプリの入口。ゲーム開始・ユーザー登録・ログイン状態の確認。 |
| 表示 | **ゲームスタート**ボタン、**ユーザー登録**（またはユーザー画面への導線）ボタン。 |
| ログイン時 | 「ログイン中」であることが分かる表示、および**ログイン中ユーザーのポイント**表示（要件 4.2 と整合）。 |
| 遷移 | ゲームスタート → Game 画面。ユーザー登録導線 → User 画面。 |

### 6.2 Game（ゲーム）

| 項目 | 内容 |
|------|------|
| 目的 | ブラックジャックをプレイする。 |
| 入り口 | Top の**ゲームスタート**から遷移。 |
| 表示・操作 | 手札、合計（プレイヤー側は分かりやすく）、ヒット / スタンド等の操作 UI。ディーラーはコンピューターとして自動進行。 |
| 終了時 | 結果表示の後、**Top 画面へ戻る**（リダイレクトまたは明示的な「トップへ」導線）。 |

### 6.3 User（ユーザー）

| 項目 | 内容 |
|------|------|
| 目的 | 新規登録、およびログイン状態でのユーザー情報表示。 |
| 入り口 | Top の**ユーザー登録**（ユーザー関連）導線から遷移。 |
| 未ログイン | **ユーザー名**・**パスワード**入力による**ユーザー登録**、および**ログイン**フォーム（同一画面内タブ切替でも、別 URL でも可）。 |
| ログイン中 | **ユーザー情報**の表示（ユーザー名、ポイント等）。ログアウト操作を提供することが望ましい。 |

## 7. データモデル（要件レベル）

実装時に Django モデルとして定義する。最低限以下を想定する。

- **ユーザー**: Django 標準のユーザーモデル拡張、または `AbstractUser` / `OneToOne` プロファイルで**ユーザー名**と**パスワード（ハッシュ保存）**、**ポイント**を保持。
- **ゲーム履歴（任意）**: 監査・表示用に、日時・結果・ポイント変動を保存するかは実装判断とする。

## 8. セキュリティ・認証

- パスワードは平文保存しない（Django 標準のハッシュ化を利用）。
- ゲーム API および Game 画面は**認証済みユーザーのみ**アクセス可能とする（未認証はリダイレクト等で拒否）。

## 9. 用語

| 用語 | 意味 |
|------|------|
| プレイヤー | ログインした人間ユーザー |
| ディーラー / コンピューター | サーバー側ロジックが操作する相手 |
| ポイント | アプリ内通貨・スコア。ユーザー単位で 1 列（または関連テーブル）で管理 |
